## What Didn't Go Well

### 1. UX Fallbacks — Gradient and Count-Up Animation Not Delivered
- `expo-linear-gradient` was not in package.json — solid color fallback used instead of gradient background for SpendingHero
- `AnimatedTextInput` count-up approach failed in test environment — simpler scale pulse used instead
- **Impact:** Dashboard works well but doesn't match the full UX spec vision
- **Decision:** GOZE decided to defer these enhancements — not needed right now

### 2. Deactivated Subscription Swipe Opacity Bug Discovered
When swiping deactivated (is_active === false) subscription items, the background action buttons (cancel, edit, delete) are visible through the card due to opacity being applied to the entire container rather than just the foreground row.
- **Root Cause:** Opacity applied to entire SwipeableSubscriptionCard container, not just foreground
- **Impact:** Visual bug affecting deactivated subscription UX
- **Priority:** HIGH — to be fixed before or during Epic 4

### 3. Hard-Coded Dates in Tests
Story 3.4 required debugging because mockSubscription `renewal_date` was hard-coded. Tests using fixed dates may break when run in the future.
- **Root Cause:** No established pattern for date mocking in tests
- **Impact:** Potential test fragility over time

### 4. Duplicate Text Match in Tests
Story 3.2 had test failures due to same price text appearing in both SpendingHero and CategoryBreakdown. Required switching from `getByText` to `getAllByText`.
- **Root Cause:** Multiple components rendering same data values
- **Pattern Note:** Use `getAllByText` when testing screens with multiple data-displaying components

### 5. Hardcoded Opacity Cleanup — 3 Epics Overdue
LOW priority item from Epic 1 retro, carried through Epic 2, still not addressed. Related to the newly discovered swipe opacity bug.

## Key Insights & Learnings

1. **Applied retro action items = measurable quality improvement** — Epic 2 had 40+ code review findings. Epic 3 had 1. The difference was consistently applying the retro commitments.

2. **Strong utility foundation enables exceptional velocity** — 4 stories in one day because Epic 2 laid solid groundwork. Investment in utilities pays compound returns.

3. **Zero dependency discipline is sustainable** — Even when chart libraries seemed necessary (Story 3.3), the team found lighter solutions. This keeps the bundle lean and reduces maintenance burden.

4. **Frontend-only epics are fast; backend epics will be different** — Epic 4 introduces completely new territory (Edge Functions, pg_cron, Push API). The team should not expect Epic 3's velocity.

5. **LOW priority carry-forwards accumulate** — The opacity issue from Epic 1 is now manifesting as a visible bug. Carrying items forward indefinitely has consequences.
