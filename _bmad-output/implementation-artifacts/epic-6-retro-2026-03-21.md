## What Went Well

1. **Thorough pre-epic preparation paid off** — `epic-6-prep-revenuecat-vs-storekit-decision.md`, `epic-6-prep-iap-guideline-review.md`, `epic-6-prep-iap-test-strategy.md`, `epic-6-prep-revenuecat-sandbox-setup.md` all prepared before Story 6.1. No mid-epic surprises on architecture or compliance.
2. **Zero production incidents** — 5 stories delivered cleanly. No regressions across 608 tests.
3. **Security bug caught in review (6.3 H2)** — Android Google Play validation was returning `valid=true` always when credentials not configured. Would have allowed free premium access. Caught in code review before production.
4. **Clean foundation for each story** — Stories built incrementally on each other. 6.4 and 6.5 had clear integration points because 6.1–6.3 laid solid groundwork.
5. **Expo Go compatibility solved cleanly** — Metro-level IAP mock preserves Expo Go development workflow without affecting production `android`/`ios` builds.
6. **4-day delivery** — Full monetization infrastructure (freemium gate, paywall, purchase, restore, entitlement enforcement) in 4 days with code review on every story.

---

## What Didn't Go Well / Challenges

1. **Library native module surprise (again)** — `react-native-iap` v14+ uses `react-native-nitro-modules` which crashes Expo Go. Discovered at implementation time (6.3), not during pre-research. Pattern from Epic 5 (library API surprises) repeated. The library validation checkpoint exists but did not include Expo Go/native module compatibility check.

2. **Grace period bug designed in too late** — The offline/network failure case for `checkPremiumStatus()` was not specified until Story 6.5. The bug (immediate downgrade on network failure) was written in 6.3 and persisted for 2 stories. For money-related flows, failure scenarios must be explicit ACs — not left to implementation.

3. **`usePremiumStore` grew large** — Every story added state and actions to the store. By 6.5 it carries: `isPremium`, `planType`, `expiresAt`, `purchaseInProgress`, `restoreInProgress`, `purchaseErrorMessage`, `lastValidatedAt`, retry logic, `isFeatureAvailable()`, `canAddSubscription()`. Manageable now but a signal to watch.

---

## Key Insights & Lessons Learned

1. **"We didn't know" is not valid for money-related flows** — `context7`, `brave-search` MCP servers exist for pre-implementation research. Any dependency touching payment, bank, or financial data must be researched before the story starts, not during.

2. **Network/offline failure cases must be explicit ACs** — Especially for external API calls (IAP, bank aggregator). "What happens offline?" must appear in story spec, not discovered in a later story's Dev Notes.

3. **Library validation checkpoint needs Expo Go/native module question** — Add: _"Does this library use native modules? Is it compatible with Expo Go managed workflow?"_ to the pre-story checklist.

4. **Code review is non-negotiable for payment flows** — 3 stories had HIGH findings. One was a security vulnerability. Review process proved its value every time.

5. **GOZE's standard for money flows: zero tolerance for mistakes** — The team must approach IAP, bank integration, and any financial feature with higher rigor than other epics.
