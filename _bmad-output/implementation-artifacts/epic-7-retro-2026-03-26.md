## Debug Session Notes — 2026-03-31

### Tink `REQUEST_FAILED_CREATE_AUTHORIZATION_CODE` — Root Cause Found

**Status:** Root cause confirmed. Fix pending Tink Console approval.

#### Root Cause: Continuous Access Not Enabled

The `REQUEST_FAILED_CREATE_AUTHORIZATION_CODE` error is caused by **"Continuous Access" (Permanent Users) not being enabled at the Tink Console app level**. This is a product-level feature that requires explicit approval from Tink — it cannot be enabled by configuring API calls alone.

- Without Continuous Access approval, the permanent user flow (`authorization_code` in Tink Link URL) will always fail at the callback authorization code creation step, regardless of scope or user_id configuration.
- The one-time flow (no `authorization_code`) continues to work in sandbox.
- **Tink support request submitted: 2026-03-31.** Awaiting approval via email.

#### To Activate When Approval Arrives

1. Go to Tink Console → your app → **Transactions → Data access**
2. Click **"Send request"** if not already done
3. Once approved (email confirmation), the permanent user flow will work in sandbox
4. Test with demo bank (`de-demobank-open-banking-redirect`) using a real SubTrack account

#### What Was Investigated & Fixed

During this session, several issues were found and resolved independent of the Continuous Access blocker:

**1. `tink-link-session` edge function — complete rewrite (deployed 2026-03-31)**

- **Removed** `lookupTinkUserId` function — was calling `GET /api/v1/user?external_user_id=...` which returns 403 with a client token. This was the `[step:user_lookup] Tink API 403` error in logs.
- **Introduced `TinkUserRef` discriminated union type** to properly handle user identity:
  - `{ id: tinkUserId, type: 'user_id' }` — when `/user/create` returns 201 (new user), Tink's internal `user_id` is parsed and used in the delegate call
  - `{ id: externalUserId, type: 'external_user_id' }` — when `/user/create` returns 409 (existing user) without a `user_id` in the body, falls back to Supabase user ID as `external_user_id`
- **Confirmed:** `external_user_id` is fully interchangeable with `user_id` in the `/api/v1/oauth/authorization-grant/delegate` endpoint (official Tink API docs)
- **Added data scopes to delegate call:** `accounts:read,balances:read,transactions:read` were missing from the original delegate scope. These are required for `transactions/connect-accounts` product — without them, Tink cannot create the callback authorization code post-bank-auth.
- Final delegate scope: `authorization:read,credentials:read,credentials:refresh,credentials:write,providers:read,user:read,provider-consents:read,provider-consents:write,accounts:read,balances:read,transactions:read`

**2. New user `is_premium=true` bug — fixed**

- **Root cause:** Zustand `persist` middleware stores `isPremium` in AsyncStorage. When a user logs out and a new user logs in, the persisted state from the previous user bleeds into the new session.
- **Fix 1 (`usePremiumStore.ts`):** When `user_settings` row returns `null` (no row), explicitly reset `isPremium: false` instead of leaving persisted state untouched.
- **Fix 2 (`AuthProvider.tsx`):** Added `usePremiumStore.setState({ isPremium: false, ... })` inside the `SIGNED_OUT` auth event handler to immediately clear premium state on logout.

**3. `user_settings` not auto-created for new users — fixed**

- **Root cause:** `user_settings` row was only created when the user changed a setting (e.g., calendar preferences). New users had no row, so `checkPremiumStatus` returned `null` data.
- **Fix:** Created and pushed Supabase migration `20260331000000_auto_create_user_settings.sql` — PostgreSQL trigger `on_auth_user_created_settings` now auto-inserts a `user_settings` row on every new `auth.users` insert via `handle_new_user_settings()` function.

**4. Dev-only premium bypass — added**

- **Context:** In sandbox, IAP "failed to start purchase" in dev builds (expected — real IAP unavailable in Metro). Needed a way to test premium-gated features without a real purchase.
- **Fix:** Added `EXPO_PUBLIC_DEV_FORCE_PREMIUM=true` to `.env`. In `usePremiumStore.checkPremiumStatus`, guarded by `__DEV__ && process.env.EXPO_PUBLIC_DEV_FORCE_PREMIUM === 'true'` — has zero effect in production builds because `__DEV__` is always `false` in EAS/production.
- Added `DEV_FORCE_PREMIUM: boolean` to `src/config/env.ts` Env interface.

## Debug Session Notes — 2026-04-16

### Android Launch Crash — RESOLVED ✅

**Error:** `[runtime not ready]: Error: Missing environment variable: EXPO_PUBLIC_SUPABASE_URL`

**Root cause:** `src/config/env.ts` used a `getEnvVar(key: string)` helper that accessed `process.env[key]` with a **dynamic key**. `babel-preset-expo`'s `inline-env-vars` plugin only inlines `EXPO_PUBLIC_*` vars when accessed via **static member expressions** (`process.env.EXPO_PUBLIC_SUPABASE_URL`). Dynamic access (`process.env[key]`) is never inlined — the value stays `undefined` at runtime in Hermes.

**Why it was hard to find:** EAS env vars were correctly configured (`eas.json` `env` section), `.env.production` was tried, EAS Environment Variables were set — all irrelevant, because none of them can fix dynamic key access. The Hermes binary was verified to confirm the URL string was absent from the bundle across all builds.

**Fix:** Rewrote `env.ts` to use direct static access:
```typescript
// Before (broken — dynamic key, never inlined):
function getEnvVar(key: string): string { return process.env[key]; }
SUPABASE_URL: getEnvVar('EXPO_PUBLIC_SUPABASE_URL')

// After (correct — static access, inlined by Babel):
SUPABASE_URL: requireEnv(process.env.EXPO_PUBLIC_SUPABASE_URL, 'EXPO_PUBLIC_SUPABASE_URL')
```

**Files changed:**
- `src/config/env.ts` — replaced `getEnvVar(key)` with `requireEnv(value, name)` using static `process.env.EXPO_PUBLIC_*` access
- `eas.json` — added all `EXPO_PUBLIC_*` vars to production `env` section; added `"cache": { "disabled": true }` (kept for safety)

**Verification:** AAB bundle inspected via `grep -ac "xdvlhtslslglbdxvhhgn"` before Play Console upload — URL confirmed present. App launched successfully on real Android device from Play Store internal track. ✅

### Biometric Login — "Biometric authentication failed" (FIXED ✅)

**Status:** ✅ Root cause found and fixed (2026-04-16 session)

**Root cause:** Double biometric gating on Android. `enrollBiometric` stored the refresh token with `ACCESS_CONTROL.BIOMETRY_ANY` in `react-native-keychain`. On Android, this forced a second biometric prompt when `getGenericPassword` was called, conflicting with the `react-native-biometrics` `simplePrompt` that had just succeeded. The second prompt threw an error that didn't match any known patterns, falling to the generic "Biometric authentication failed" catch-all.

**Fix:**
1. Removed `accessControl: Keychain.ACCESS_CONTROL.BIOMETRY_ANY` from `enrollBiometric` — `simplePrompt` is the sole biometric gate, Keychain is secured by device lock (`WHEN_PASSCODE_SET_THIS_DEVICE_ONLY`)
2. Added explicit try/catch for Keychain access in `authenticateWithBiometric` — shows actionable "disable and re-enable biometrics" message
3. Improved error mapping: added Android-specific patterns (`negative button`, `keystore`, `cipher`, `invalidated`, `lockout`)

## Debug Session Notes — 2026-04-16 (Session 2)

### Biometric "Session Expired" — Root Cause Found & Fixed ✅

**Status:** ✅ Fixed (2026-04-16)

**Root cause:** Supabase rotates refresh tokens on every use. `enrollBiometric` stores a snapshot of the current refresh token in Keychain. When Supabase silently refreshes the session (on app startup, app foreground, TOKEN_REFRESHED event), the refresh token changes. The Keychain snapshot becomes stale. On the next biometric login attempt, `supabase.auth.setSession({ refresh_token: stale_token })` fails → "Session expired. Please log in with your password."

**Fix:** `AuthProvider.tsx` now updates the Keychain with the latest refresh token on every `SIGNED_IN` and `TOKEN_REFRESHED` event when `isBiometricEnabled` is true. This keeps the biometric credential in sync with Supabase's token rotation.

**Files changed:**
- `src/app/providers/AuthProvider.tsx` — import `enrollBiometric`, call it on `SIGNED_IN`/`TOKEN_REFRESHED` events when biometric enabled

### Bank Connection "Not Connected Yet" — Root Cause Found & Fixed ✅

### Tink "Scan for Subscriptions" Kills Active Bank Connection — FIXED ✅

**Status:** ✅ Fixed (2026-04-20)

**Root cause:** `tink-detect-subscriptions` edge function attempts server-side `authorization-grant` to get a user access token for rescanning. ### Biometric "Session Expired" After Token Rotation — FIXED ✅ (Second fix)

**Status:** ✅ Fixed (2026-04-20) — first fix (2026-04-16) had a silent failure path

**Root cause (2026-04-16 fix regression):** The April 16 fix called `enrollBiometric(session.refresh_token)` in `AuthProvider` on every `SIGNED_IN`/`TOKEN_REFRESHED` event. However, `enrollBiometric` starts with `rnBiometrics.createKeys()`. On Android, `createKeys()` throws when RSA keys already exist in the Keystore. This exception was caught by the outer try/catch in `enrollBiometric`, returning `{ success: false }` before `Keychain.setGenericPassword` was ever called. The `.catch(() => {})` in `AuthProvider` only catches Promise rejections, not the `success: false` return — so the failure was completely invisible. Result: after initial enrollment, the keychain token was NEVER updated on subsequent token rotations → stale token → "Session expired" on next biometric login.

**Fix:** Added `updateBiometricToken(refreshToken)` to `biometricService.ts` — updates ONLY the Keychain value without calling `createKeys()`. `AuthProvider` now calls `updateBiometricToken` instead of `enrollBiometric` for token sync. `enrollBiometric` (which calls `createKeys()`) is still used only when the user first enables biometrics.

**Files changed:**
- `src/features/auth/services/biometricService.ts` — new `updateBiometricToken()` function
- `src/app/providers/AuthProvider.tsx` — import `updateBiometricToken`, replace `enrollBiometric` call in token sync handler
